## トリガーと引数解析
- コマンドフォーマット: `/pr [base_branch]`
- `base_branch` (オプション): PRのターゲットとなるベースブランチ。
  - デフォルトは `main` とします。
  - 指定された場合はそのブランチを使用し、リモートまたはローカルに存在するか事前に確認してください（`git ls-remote` または `git show-ref`）。

## 実行手順（ステップ・バイ・ステップで実行すること）

### 1. 現在の状態と変更内容の把握
以下のコマンドを順次実行し、ワークスペースの状態を完全に把握してください。
- リポジトリルートの確認: `git rev-parse --show-toplevel`
- 現在のブランチ確認: `git branch --show-current`
- ステータス確認: `git status --short`
- 差分の詳細確認: `git diff --cached` および `git diff`
- 未追跡ファイルの確認: `git ls-files --others --exclude-standard`

### 2. 変更内容の分析とコミットの設計（思考フェーズ）
取得した `git diff` やファイル内容を読み込み、以下の分析を行ってください。
- **論理的なグループ化**: 変更を「1コミット = 1つの論理的な変更」となるように分類します。（例: 認証機能の追加と、無関係なUIの修正は分ける）
- **未追跡ファイルの判定**: 新規追加されたファイルが今回のPRのスコープに含まれるべきか判断します。

### 3. 適切な粒度でのコミット作成
設計したグループごとに、以下の形式でコミットを作成してください。
- コミットメッセージ形式: `{type}: {簡潔な説明（日本語50文字以内）}`
  - `type`の例: `feat`, `fix`, `docs`, `refactor`, `test`, `chore`
- 実行手順: 対象ファイルを `git add` し、`git commit -m "..."` を実行。
- 確認: 毎コミット後、`git log --oneline -1` で意図通りか確認してください。

### 4. 仕様書（ドキュメント）の整合性チェックと更新
コードの変更内容に基づいて、仕様書と乖離が発生していないか確認し、必要なら更新します。
- **対象ファイル**:
  - `documents/requirements.md` (UIや機能要件の変更)
  - `documents/api-specification.md` (LambdaやAPIエンドポイントの変更)
  - `documents/architecture.md` (Terraformやインフラ構成の変更)
- **アクション**: 対象の仕様書を読み込み、更新が必要な場合はファイルを編集します。
- **コミット**: 仕様書を更新した場合は、必ず独立したコミットを作成してください。（例: `docs: API仕様書に新規エンドポイントを追記`）

### 5. リモートへのプッシュ
- リモートブランチの有無を確認: `git ls-remote --heads origin {current_branch}`
- 存在しない場合: `git push -u origin {current_branch}`
- 存在する場合: `git push`

### 6. PRタイトルと本文の自動生成（厳格な文字数制限）
ベースブランチとの差分（`git log --oneline {base_branch}..HEAD` および `git diff --stat`）を分析し、以下のルールに従ってテキストを生成します。

**【PRタイトル】**
- 形式: `{type}: {最も重要な変更の簡潔な説明}`
- 制限: 日本語50文字以内。

**【PR本文】**
- **制限**: 全体で200〜400文字程度。長文は避け、要点のみを記載すること。（技術的詳細はコミットメッセージに委ねる）
- **フォーマット**:
  ```markdown
  ## 変更内容
  - （2〜3行で簡潔に、目的と実際の変更内容を記載）
  
  ## 変更の詳細
  - （主な変更点を3〜5項目の箇条書きで1行ずつ記載）
  
  ## 影響範囲
  - （影響を受ける機能やモジュールを記載。なければ「特になし」）
  
  ## 確認事項
  - （レビュワーに特に見てほしいポイントがあれば記載）
  ```

### 7. PRの作成 (GitHub CLI)
生成したタイトルと本文を使用して、gh コマンドでPRを作成します。

実行コマンド例: gh pr create --base {base_branch} --title "タイトル" --body "本文"

実行後、発行されたPRのURLをユーザーに報告してタスクを完了します。

### ⚠️ エージェントへの重要注意事項
推測の禁止: 変更の目的はファイルの内容とDiffから実際に読み取って理解してください。

手動介入の排除: PR本文やタイトルは、あなたの分析結果から完全に自動生成し、途中でユーザーにテキストの入力を求めないでください。
